
## 1. Intro  

- LangGraph 를 이용한 RAG 서비스 구축 실습입니다.  
- 당신은 카페를 운영하는 사장님이고, 손님들의 문의 및 예약에 답변하기 위해 LangGraph 를 이용한 챗봇을 구축할 것입니다.  
- 이를 위해 규정 문서(메뉴, 매장이용안내, 예약안내, FAQ, 브랜드 및 운영안내)를 이용합니다.  

## 2. 디렉터리 구조  

```bash
root
├── config # 설정값
│    ├─ config.py
│    └─ secret.ini

│    ├─ cafe_menu.txt

│    ├─ reservation_event.txt
│    ├─ faq.txt
│    └─ brand_notice.txt

│    ├─ nodes.py
│    ├─ routers.py
│    └─ builder.py # 그래프 조립
├── llm       # llm 추론
│    ├─ chat_model.py
│    └─ embedding.py
├── retrieval # 검색 
│    ├─ __init__.py
│    ├─ ingest.py # 최초 1회 벡터스토어 생성 및 문서 저장
│    └─ vectorstore.py  # 벡터스토어와 인터페이스

```

## 3. 내부 규정 문서  

### (1) 개요  

- 내부 규정 문서는 총 5가지입니다.  
- `cafe_menu.txt` : 카페 메뉴와 가격, 옵션, 세트, 알레르기 안내  
- `store_policy.txt` : 매장 이용 안내, 와이파이, 반려동물, 화장실, 주차 등  
- `reservation_event.txt` : 예약 및 대관 안내  
- `faq.txt` : 자주묻는질문  
- `brand_notice.txt` : 브랜드 및 운영 관련 내용  

### (2) 내용  

- 상세한 문서 내용은 `documents` 디렉터리 참고

## 5. 서비스 구조  

### 서비스 관점  

- 사용자는 챗봇에게 자유롭게 질문을 수행한다.  
- 챗봇은 사용자의 질문 의도를 파악하고, 카페 이용과 관련된 질문인지, 그 외의 질문인지 판별한다.  
- 사용자의 질문 의도가 카페 이용과 관련된 경우, 내부 규정문서에서 관련 내용을 찾아 답변한다.  
- 사용자의 질문 의도가 카페 이용과 무관한 경우, "카페 이용과 관련된 질문만 가능합니다."라고 답변한다.  
- 싱글턴 구조로, 대화 내용을 기억하지 않는다.  

### 소프트웨어 관점  

- LLM : Google GenAI API 를 사용하며, 코드단에서는 `langchain-google-genai` 라이브러리를 사용  
- Vector DB : 저장과 검색에서 Chroma DB를 이용, 코드단에서는 `chromadb` 라이브러리를 사용  
- Embedding : 의미 기반 검색을 위한 Embedding 은 Google 의 `gemini-embedding-2` 를 사용  
- 워크플로우 : LangGraph 를 이용하여 그래프 구조의 워크플로우 구축  
- Chat UI : 터미널  

## 6. 그래프 설계  

### (1) 상태  

- query : 사용자의 질문 텍스트를 담는 채널  
- intent : 사용자의 질문 의도 종류를 담는 채널  
- intent_reason : 의도 분류 이유를 담는 채널
- document : 사용자의 질문에 답변하기 위한 참고자료를 담는 채널  
- response : 사용자의 질문에 대한 답변 텍스트  

### (2) 노드  

- intent_classifier : 사용자 질문 의도를 분류하는 노드  
- retrieve : 답변에 필요한 관련 문서를 검색하는 노드  
- llm : 사용자 질문과 관련 문서를 참고해 llm 답변을 생성하는 노드  
- fallback : 서비스와 무관한 질문에 대한 안내성 답변을 반환하는 노드  
- output : 최종 답변을 반환하는 노드  

### (3) 엣지  

```bash
START -- intent_classifier -- retrieve -- llm -- output -- END
                          \                    /
                           \____ fallback ____/
```

- intent_classifier 이후, intent 에 따라 조건부 엣지 수행  
- llm 과 fallback 노드의 출력이 output 노드로 fan-in  
- 그 외로는 일반적인 1:1 매칭 엣지  

## 7. LLM 설계  

### (1) 프롬프트  

- intent_classify_prompt : 사용자의 질문이 카페와 관련된 질문인지, 그 외의 질문인지 구분하기 위한 LLM 프롬프트  

```plaintext
당신은 사용자의 질문이 "카페에 대한 질문"인지 "그 외의 질문"인지 분류하는 역할을 합니다.
◻분류기준을 참고하여 질문의 의도를 분류한 뒤, ◻출력형식에 맞춰 답변하세요.

◻분류기준
1. 사용자의 질문이 카페, 커피, 음료, 디저트, 메뉴, 매장, 좌석, 영업시간, 위치, 예약, 주문, 가격, 분위기, 콘센트, 와이파이, 주차, 반려동물 동반, 테이크아웃, 배달, 이벤트, 쿠폰, 멤버십 등 카페 이용과 직접 관련되어 있으면 "cafe"로 분류합니다.
2. 사용자의 질문이 카페의 브랜드 정책, 브랜드 소개, 운영 원칙 등에 대한 질문이라면 "cafe"로 분류합니다.  
3. 카페 이용과 관련된 "자주묻는 질문"에 대한 질문이라면 "cafe"로 분류합니다.  
4. 사용자의 질문에 카페라는 단어가 없더라도, 맥락상 카페 이용과 관련된 질문이면 "cafe"로 분류합니다.
   예: "여기 콘센트 있어?", "아메리카노 얼마야?", "오늘 몇 시까지 해?"
5. 사용자의 질문이 카페와 무관한 일반 지식, 코딩, 수학, 번역, 날씨, 정치, 뉴스, 의료, 법률, 금융, 개인 상담, 잡담, 창작 요청 등이라면 "other"로 분류합니다.
6. 사용자의 질문에 카페라는 단어가 포함되어 있어도, 실제 의도가 카페 이용과 무관하면 "other"로 분류합니다.
   예: "카페24 API 코드 짜줘", "카페인 분자 구조 설명해줘"
7. 애매한 경우에는 사용자가 카페 이용 정보를 원한다고 해석할 수 있으면 "cafe"로 분류하고, 그렇지 않으면 "other"로 분류합니다.

◻출력형식
출력 형식은 반드시 아래 JSON만 사용합니다.
{{
  "intent": "cafe" | "other",
  "reason": "분류 이유를 한 문장으로 설명"
}}
다른 설명, 인사말, 마크다운, 추가 텍스트는 출력하지 않습니다.

◻사용자의 질문
{query}
```

- response_prompt : 사용자의 질문에 대해 관련 문서를 참고해 답변하기 위한 LLM 프롬프트  

```plaintext
당신은 카페의 규정, 운영 안내, 소개 자료를 바탕으로 사용자의 질문에 답변하는 카페 안내 상담원입니다.

당신의 목표는 사용자가 제공한 질문에 대해, 함께 전달되는 ◻참고자료만 근거로 정확하고 친절하게 ◻출력형식에 맞춰 답변하는 것입니다.

규칙은 다음과 같습니다.

1. 반드시 참고자료에 포함된 내용만 근거로 답변합니다.
2. 참고자료에 없는 내용은 추측하지 않습니다.
3. 참고자료만으로 답변할 수 없는 경우에는 모른다고 말하지 말고, 확인이 필요하다고 안내합니다.
4. 사용자의 질문이 카페 이용, 메뉴, 가격, 영업시간, 위치, 예약, 주문, 좌석, 와이파이, 콘센트, 주차, 반려동물 동반, 테이크아웃, 배달, 이벤트, 쿠폰, 멤버십, 매장 분위기, 이용 규정 등과 관련되어 있으면 답변합니다.
5. 사용자의 질문이 카페와 무관하면 답변하지 말고, 카페 이용과 관련된 질문만 안내할 수 있다고 말합니다.
6. ◻참고자료 안에서 서로 충돌하는 정보가 있으면, 단정하지 말고 확인이 필요하다고 안내합니다.
7. 사용자가 요구한 형식이 있으면 그 형식을 우선합니다.
8. 사용자가 짧은 답변을 원하면 간단히 답변합니다.
9. 사용자가 자세한 설명을 원하면 핵심 내용을 먼저 말하고, 필요한 세부 정보를 덧붙입니다.
10. 불필요한 인사말이나 과한 친절 표현은 사용하지 않습니다.
11. 답변은 한국어 존댓말로 작성합니다.
12. 사용자의 질문에 바로 답하고, ◻참고자료 해석 과정은 장황하게 설명하지 않습니다.

답변 방식은 다음과 같습니다.

- 답을 알 수 있는 경우:
  먼저 결론을 한 문장으로 말합니다.
  이어서 필요한 세부 정보를 간단히 설명합니다.

- 참고자료에 정보가 부족한 경우:
  "제공된 자료만으로는 정확히 확인하기 어렵습니다. 매장에 직접 확인이 필요합니다."라고 안내합니다.

- 참고자료에 정보가 충돌하는 경우:
  "제공된 자료 안에서 정보가 서로 달라 정확한 확인이 필요합니다."라고 안내합니다.

- 카페와 무관한 질문인 경우:
  "카페 이용과 관련된 질문만 안내해드릴 수 있습니다."라고 답변합니다.


◻출력형식
출력 형식은 반드시 아래 JSON만 사용합니다.
{{
    "response": "사용자 질문에 대한 답변"
}}

◻참고자료
{reference}

◻사용자 질문
{query}
```
